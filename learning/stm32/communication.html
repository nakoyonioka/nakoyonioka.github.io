Module 3: Communication Protocols

UART, SPI, and I2C Interfaces

Microcontrollers rarely work in isolation. They need to communicate with other devices, such as sensors, displays, other microcontrollers, or even computers. This communication happens through various communication protocols, each with its own advantages and use cases.

In this module, you'll get an overview of three common serial communication protocols: UART (Universal Asynchronous Receiver/Transmitter), SPI (Serial Peripheral Interface), and I2C (Inter-Integrated Circuit). You'll learn their basic principles and how to configure them in STM32CubeIDE using the HAL library.

What You'll Need:

  • An STM32 Development Board
  • STM32CubeIDE installed
  • (Optional, but recommended for hands-on)
    • USB-to-UART converter (e.g., FTDI, CP2102)
    • Small OLED display (e.g., SSD1306 I2C OLED)
    • SPI-compatible sensor (e.g., MPU6050 accelerometer/gyroscope)
    • Jumper wires, breadboard

1. UART (Universal Asynchronous Receiver/Transmitter):

UART is a simple, two-wire (TX/RX) asynchronous serial communication protocol. It's commonly used for debugging (like Arduino's Serial Monitor) or communicating with modules like Bluetooth, GPS, or Wi-Fi.

Configuration in STM32CubeIDE:

  1. Open your `.ioc` file.
  2. Go to "Connectivity" and select "USART1" (or any available USART/UART).
  3. Set "Mode" to "Asynchronous".
  4. In the "Parameter Settings" tab, configure:
    • Baud Rate: Common values are 9600, 115200.
    • Word Length: 8 Bits.
    • Parity: None.
    • Stop Bits: 1.
  5. Generate code by saving the `.ioc` file.
  6. In `main.c`, you can use HAL functions to send/receive data: 
    
/* USER CODE BEGIN 0 */
#include <string.h> // For strlen()
/* USER CODE END 0 */

// ... inside while(1) loop ...
/* USER CODE BEGIN 3 */
  char msg[] = "Hello from STM32 via UART!\r\n";
  HAL_UART_Transmit(&huskart1, (uint8_t*)msg, strlen(msg), HAL_MAX_DELAY);
  HAL_Delay(1000);
/* USER CODE END 3 */

    Connect your STM32's configured UART TX pin to the RX of a USB-to-UART converter, and STM32's RX to the TX of the converter. Connect GNDs. Use a serial terminal program on your computer (e.g., PuTTY, CoolTerm) to see the messages.

2. SPI (Serial Peripheral Interface):

SPI is a synchronous, full-duplex, master-slave communication protocol that uses four wires: SCK (Clock), MOSI (Master Out, Slave In), MISO (Master In, Slave Out), and SS/CS (Slave Select/Chip Select). It's faster than UART and commonly used for displays, SD cards, and some sensors.

Configuration in STM32CubeIDE:

  1. Open your `.ioc` file.
  2. Go to "Connectivity" and select "SPI1" (or any available SPI).
  3. Set "Mode" to "Full-Duplex Master".
  4. Configure the "Parameter Settings" (e.g., "Frame Format", "Data Size", "Prescaler").
  5. Identify the pins for SCK, MOSI, MISO, and a GPIO output for CS.
  6. Generate code by saving the `.ioc` file.
  7. Example SPI transmit (assuming `hspi1` is your SPI handle): 
    
/* USER CODE BEGIN 3 */
  uint8_t tx_data[] = {0xAA, 0xBB, 0xCC};
  // Pull CS low to select the slave
  HAL_GPIO_WritePin(CS_GPIO_Port, CS_Pin, GPIO_PIN_RESET);
  HAL_SPI_Transmit(&hspi1, tx_data, sizeof(tx_data), HAL_MAX_DELAY);
  // Pull CS high to deselect the slave
  HAL_GPIO_WritePin(CS_GPIO_Port, CS_Pin, GPIO_PIN_SET);
  HAL_Delay(500);
/* USER CODE END 3 */

3. I2C (Inter-Integrated Circuit):

I2C is a two-wire, multi-master, multi-slave serial bus. It uses SDA (Serial Data Line) and SCL (Serial Clock Line). It's slower than SPI but requires fewer pins and is great for connecting multiple low-speed peripherals like EEPROMs, real-time clocks (RTCs), and many sensors.

Configuration in STM32CubeIDE:

  1. Open your `.ioc` file.
  2. Go to "Connectivity" and select "I2C1" (or any available I2C).
  3. Set "Mode" to "I2C".
  4. Configure "Parameter Settings" (e.g., "Speed Mode" to "Fast Mode" for 400kHz).
  5. Generate code by saving the `.ioc` file.
  6. Example I2C write (assuming `hi2c1` is your I2C handle, `slave_address` is 7-bit address shifted left by 1): 
    
/* USER CODE BEGIN 3 */
  uint8_t slave_address = 0x3C << 1; // Example: OLED display address
  uint8_t data_to_send[] = {0x00, 0xAF}; // Example: OLED command to turn on display

  HAL_StatusTypeDef status = HAL_I2C_Master_Transmit(&hi2c1, slave_address, data_to_send, sizeof(data_to_send), HAL_MAX_DELAY);
  if (status != HAL_OK) {
    // Handle error (e.g., toggle an LED)
    HAL_GPIO_TogglePin(LD2_GPIO_Port, LD2_Pin);
  }
  HAL_Delay(1000);
/* USER CODE END 3 */

Troubleshooting Tips:

  • UART:
    • Baud rate mismatch is the most common issue.
    • Cross-check TX/RX connections (STM32 TX to converter RX, STM32 RX to converter TX).
    • Ensure GNDs are connected.
  • SPI:
    • Incorrect clock polarity/phase (CPOL/CPHA) settings. Refer to the slave device's datasheet.
    • CS pin not being correctly toggled (pulled low before transmit, high after).
    • MISO/MOSI swapped.
  • I2C:
    • Missing pull-up resistors on SDA and SCL lines (usually 4.7kÎ©).
    • Incorrect slave address (remember it's often 7-bit, and you might need to shift it left by 1 for HAL functions).
    • Device not powered or not connected correctly.
    • Bus busy errors (reset the device or MCU).

You've now gained an understanding of three essential communication protocols for STM32 microcontrollers. Mastering these protocols will enable your embedded projects to interact with a vast array of external components and systems, opening up endless possibilities for complex and intelligent designs.
